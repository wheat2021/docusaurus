# 创建文档

创建一个 Markdown 文件，`greeting.md`，并将其放置在 `docs` 目录下。

```bash
website # 你的网站根目录
├── docs
│   └── greeting.md
├── src
│   └── pages
├── docusaurus.config.js
├── ...
```

```md
---
description: 创建一个内容丰富的文档页面。
---

# 来自 Docusaurus 的问候

你准备好为你的开源项目创建文档网站了吗？

## 标题

将显示在右上角的目录中

这样你的用户无需向下滚动或甚至无需阅读太多内容就能知道这个页面是关于什么的。

## 默认情况下，只有 h2 和 h3 会出现在目录中。

你可以在每个文档中或在主题配置中配置目录的标题级别。

标题间距适中，层次清晰。

- 列表将帮助你
- 展示你希望用户记住的
- 关键点
  - 你可以嵌套它们
    - 多次
```

:::note

`docs` 目录下所有以下划线（`_`）为前缀的文件都被视为“部分”页面，默认将被忽略。

阅读更多关于[导入部分页面](../markdown-features/markdown-features-react.mdx#importing-markdown)的内容。

:::

## 文档 front matter {#doc-front-matter}

[Front matter](../markdown-features/markdown-features-intro.mdx#front-matter) 用于为你的文档页面提供额外的元数据。Front matter 是可选的——Docusaurus 能够在没有 front matter 的情况下推断出所有必要的元数据。例如，下面介绍的[文档标签](#doc-tags)功能需要使用 front matter。有关所有可能的字段，请参阅[API 文档](../../api/plugins/plugin-content-docs.mdx#markdown-front-matter)。

## 文档标签 {#doc-tags}

标签在 front matter 中声明，除了[文档侧边栏](./sidebar/index.mdx)之外，还引入了另一个分类维度。

可以内联定义标签，也可以引用在[`标签文件`](../../api/plugins/plugin-content-docs.mdx#tags-file)（可选，通常是 `docs/tags.yml`）中预定义的标签。

在以下示例中：

- `docusaurus` 引用了在 `docs/tags.yml` 中声明的预定义标签键
- `Releases` 是一个内联标签，因为它不存在于 `docs/tags.yml` 中

```md title="docs/my-doc.md"
---
tags:
  - Releases
  - docusaurus
---

# 标题

内容
```

```yml title="docs/tags.yml"
docusaurus:
  label: 'Docusaurus'
  permalink: '/docusaurus'
  description: '与 Docusaurus 框架相关的文档'
```

:::tip

标签也可以用 `tags: [Demo, Getting started]` 来声明。

阅读更多关于所有可能的 [Yaml 数组语法](https://www.w3schools.io/file/yaml-arrays/)。

:::

## 组织文件夹结构 {#organizing-folder-structure}

`docs` 文件夹下 Markdown 文件的排列方式可以对 Docusaurus 内容生成产生多种影响。然而，其中大部分都可以与文件结构解耦。

### 文档 ID {#document-id}

每个文档都有一个唯一的 `id`。默认情况下，文档 `id` 是文档相对于根 docs 目录的名称（不含扩展名）。

例如，`greeting.md` 的 ID 是 `greeting`，而 `guide/hello.md` 的 ID 是 `guide/hello`。

```bash
website # 你的网站根目录
└── docs
   ├── greeting.md
   └── guide
      └── hello.md
```

然而，`id` 的**最后一部分**可以由用户在 front matter 中定义。例如，如果 `guide/hello.md` 的内容定义如下，其最终 `id` 是 `guide/part1`。

```md
---
id: part1
---

Lorem ipsum
```

在手写侧边栏，或使用与文档相关的布局组件或钩子时，ID 用于引用文档。

### 文档 URL {#doc-urls}

默认情况下，文档的 URL 位置派生自[文档 `id`](#document-id)，而后者又基于文档的文件路径。

如果文件命名为以下之一，文件名将不会包含在 URL 中：

- 命名为 `index`（不区分大小写）：`docs/Guides/index.md`
- 命名为 `README`（不区分大小写）：`docs/Guides/README.mdx`
- 与父文件夹同名：`docs/Guides/Guides.md`

在所有情况下，默认的 `slug` 都将只是 `/Guides`，不带 `/index`、`/README` 或重复的 `/Guides` 段。

:::note

此约定与[分类索引约定](./sidebar/autogenerated.mdx#category-index-convention)完全相同。然而，`isCategoryIndex` 配置并_不_影响文档 URL。

:::

使用 `slug` front matter 来提供一个明确的文档 URL 并覆盖默认的。

例如，假设你的网站结构如下：

```bash
website # 你的网站根目录
└── docs
    └── guide
        └── hello.md
```

默认情况下，`hello.md` 将在 `/docs/guide/hello` 处可用。你可以将其 URL 位置更改为 `/docs/bonjour`：

```md
---
slug: /bonjour
---

Lorem ipsum
```

`slug` 将被附加到文档插件的 `routeBasePath`，默认为 `/docs`。有关如何从 URL 中移除 `/docs` 部分，请参阅[纯文档模式](docs-introduction.mdx#docs-only-mode)。

:::note

可以使用：

- 绝对 slug：`slug: /mySlug`、`slug: /`...
- 相对 slug：`slug: mySlug`、`slug: ./../mySlug`...

:::

:::tip

更改文档的文件名或 `id`，将更改其默认 URL。为防止在重命名文件时破坏永久链接，我们建议设置一个明确的 `slug` 以保持您的 URL 稳定。

:::

#### 使文档在根目录可用

如果你想让一个文档在根目录可用，并有一个像 `https://docusaurus.io/docs/` 这样的路径，你可以使用 slug front matter：

```md
---
id: my-home-doc
slug: /
---

Lorem ipsum
```

### 侧边栏 {#sidebars}

当使用[自动生成的侧边栏](./sidebar/autogenerated.mdx)时，文件结构将决定侧边栏的结构。

我们对文件系统组织的建议是：让你的文件系统镜像侧边栏结构（这样你就不需要手写你的 `sidebars.js` 文件），并使用 `slug` front matter 来自定义每个文档的 URL。
